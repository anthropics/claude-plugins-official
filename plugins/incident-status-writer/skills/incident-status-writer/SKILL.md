# Incident Status Writer

제품팀 내부 장애 보고를 받아 **조치 완료 이후의 고객 안내용 status 공지**로 변환하는 스킬. **한국어와 영어 두 버전을 항상 함께 출력한다.**

## 입력

사용자가 보통 다음 중 하나의 형태로 내부 장애 보고를 공유한다.

- 제품팀이 사내 메신저/문서에 정리한 글 (텍스트 또는 스크린샷)
- 보통 이런 항목들을 포함: **발생 시간, 영향 범위, 원인, 조치 내용, 재발 방지 대책**
- 내부 기술 용어가 그대로 섞여 있음 (예: "tool call 히스토리 처리 로직", 특정 모델명, 내부 컴포넌트명 등)

스크린샷만 제공될 수도 있으므로, 이미지에서 위 항목들을 읽어 구조화한 뒤 작업을 시작한다.

## 출력 형식

**한국어 버전과 영어 버전을 모두 출력한다.** 각 버전은 별도의 코드 블록으로 감싸서, 사용자가 복사해 붙여넣기 쉽게 한다.

### 한국어 버전 구조

```
- 발생 현상:      ← 증상이 2개 이상일 때만 이 블록을 추가
 1) [증상1]
 2) [증상2]
- 발생 시간: YYYY-MM-DD HH:MM ~ HH:MM KST (약 N분간)
- 영향 범위: [사용자 기준으로 풀어쓴 범위]
- 원인: [쉬운 말로 바꾼 원인 설명, 1문장]
- 조치 내용: [무엇을 복구했는지 + 재발 방지 조치 언급]

[마무리 문구 - 선택] 동일한 불편을 겪으시지 않도록 모니터링을 강화하겠습니다.
```

### 영어 버전 구조

```
- Incident Time: Month Day, YYYY HH:MM ~ HH:MM (KST) (approximately N minutes)
- Affected Scope: [User-facing scope]
- Incident Details:      ← Only include this block when 2+ symptoms
 1) [Symptom 1]
 2) [Symptom 2]
- Cause: [Plain-language cause, 1 sentence]
- Resolution: [What was fixed + preventive measures]

[Closing sentence - optional] We will continue to strengthen monitoring to prevent similar issues from occurring again.
```

### 라벨 대응표 (한↔영 고정 번역)

| 한국어 | 영어 |
|---|---|
| 발생 시간 | Incident Time |
| 영향 범위 | Affected Scope |
| 발생 현상 | Incident Details |
| 원인 | Cause |
| 조치 내용 / 조치 사항 | Resolution |
| 일시적인 문제 발견 (제목) | Temporary issue detected with ... |
| 서비스 이용에 불편을 드린 점 사과드립니다 | We apologize for any inconvenience this may have caused |
| 동일한 불편을 겪으시지 않도록 모니터링을 강화하겠습니다 | We will continue to strengthen monitoring to prevent similar issues from occurring again |

### 형식 세부 규칙

**공통**
- 제목과 첫 문단(요약·정상화·사과)은 출력하지 않는다. 불릿 항목만 출력한다.
- 불릿 순서: 발생 시간 → 영향 범위 → (발생 현상) → 원인 → 조치 내용.
- 발생 현상 블록: 증상이 한 가지뿐이면 생략하고 한국어 첫 문단에서 요약. 두 가지 이상이면 1), 2) 번호로 나열.
- 마무리 문구: 체감이 큰 장애에만 추가. 경미한 건에는 생략.

**시간 표기**
- 한국어: `2026-04-20 10:25:42 ~ 11:08:41 KST (약 43분간)` 또는 `2026년 4월 20일 09:55 ~ 10:39 (약 44분간)`
- 영어: `April 20, 2026 10:25:42 ~ 11:08:41 (KST) (approximately 43 minutes)` 또는 `April 20, 2026 09:55 ~ 10:39 (approximately 44 minutes)`

## 변환 원칙 (가장 중요)

우선순위대로 적용한다.

### 1. 정중한 사과 톤 유지 (최우선)

- "조치 내용/Resolution" 뒤에는 재발 방지 약속 뉘앙스를 남긴다.
- 변명조로 들리지 않게 한다. "어쩔 수 없이", "예상치 못한", "unavoidably", "unexpected" 같은 표현은 쓰지 않는다.
- 책임 회피 표현 금지. "일부 사용자에게만", "극히 드물게", "only a few users", "in rare cases" 같은 축소 표현도 피한다.

### 2. 내부 기술 용어 완전 배제

내부 기술 용어 → 사용자 관점 언어로 반드시 치환한다. 원문에 있어도 그대로 쓰지 않는다.

| 내부 용어 (원문) | 한국어 공지용 | 영어 공지용 |
|---|---|---|
| "Claude sonnet 4.6 모델 tool call 히스토리 처리 로직" | "특정 상황에서 시스템을 지원하는 일부 서버가 일시적으로 중단되는 현상" | "a temporary disruption occurred in some servers supporting the system under specific conditions" |
| "DB 커넥션 풀 고갈" | "데이터를 불러오는 서버에 일시적으로 많은 부하가 발생" | "a temporary surge in load occurred in the server managing the data" |
| "캐시 invalidation 이슈" | "일부 정보가 실제와 다르게 표시되는 현상" | "some information was displayed differently from the actual data" |
| "특정 엔드포인트 타임아웃" | "일부 기능의 응답이 평소보다 지연됨" | "some features responded slower than usual" |
| "롤백", "배포", "deployment" | "긴급 점검 및 정상화 작업" | "emergency inspection and restoration" |
| 내부 서비스/모듈명 | 언급하지 않거나 "시스템의 특정 구성 요소" | avoid, or "a specific component of the system" |
| **[영향 범위]** "Claude sonnet 4.6 모델 사용하는 채널, ...새롭게 생성된 유저챗 또는 알프와 대화한 유저챗" | **"이슈 기간 내 새롭게 생성되어 태스크가 동작한 유저챗"** | **"user chats newly created during the incident in which tasks were triggered"** |
| **"패치 배포"**, **"앱 패치"** | **"앱을 업데이트"** | **"app was updated"** (NOT "app patch was deployed") |
| **"회귀 테스트 강화"**, **"회귀 테스트"** | **"사전 점검을 강화"** | **"Pre-release checks have been strengthened"** (NOT "Regression testing") |
| **"갇힐 수 있는 케이스"**, **"스턱되는 케이스"** | **"멈출 수 있는 상황"** | **"situations where [the feature] could get stuck"** |
| **"예상하지 못한 경로"**, **"unexpected path"** (코드 흐름 맥락) | **"예상하지 못한 경로로 진입하는 현상"** (사용자 체감 증상으로) | **"an unexpected path was triggered"** (사용자 관점 풀어쓰기) |
| **"서버 롤백"** | **"서버를 이전 상태로 복구"** | **"the server was restored to its previous state"** (NOT "rolled back") |

**원칙**: 고객은 내부 구조를 모르고 알 필요도 없다. "왜 그런 일이 일어났는지 감을 잡을 수 있는 수준"까지만 풀어서 쓴다.

예외적으로 **채널톡의 공식 기능명**은 그대로 노출한다. 단, 정식 표기를 지켜야 한다.

**대표 규칙**:
- "알프" / "Alf" / "alf" → 반드시 **ALF** (전부 대문자)
- 내부 모델명(예: "Claude sonnet 4.6", "GPT-4")은 공지에서 제거
- 내부 컴포넌트명(예: "tool call", "invoker")은 "시스템의 특정 구성 요소"로 일반화
- 채널톡 공식 기능명(태스크, 규칙, 지식, 유저챗 등)은 **풀어쓰지 말고 그대로** 사용

### 3. 사용자가 겪는 증상 중심 설명

- "원인"보다 "증상"을 먼저 명확히 한다. 사용자는 "내가 뭘 겪었던 건지"가 먼저 궁금하다.
- 내부 원인 로직이 아니라 **UI/기능 수준에서 어떻게 보였는지**로 기술한다.
  - ❌ "tool call 처리 로직 오류" / "tool call processing logic error"
  - ✅ "ALF와 대화한 답변이 평소보다 지연되어 도착" / "responses from ALF chats were delayed compared to usual"
- **"영향 범위" 작성 규칙**:
  - 기술적 단위가 아닌 사용자 단위로. "특정 리전의 pod" ❌ → "모든 채널" ✅
  - 내부 모델명/시스템명을 영향 범위 조건으로 쓰지 않는다. 예를 들어 원문이 "Claude sonnet 4.6 모델을 사용하는 채널"이라고 해도, 사용자는 자기가 어느 모델을 쓰는지 모르므로 그 조건은 제거한다.
  - "대화한 유저챗"처럼 모호한 행동 대신, **실제로 문제가 발생했던 기능 동작 단위**로 구체화한다. 예: "ALF와 대화한 유저챗" → "태스크가 동작한 유저챗" (태스크 실행이 실패한 이슈라면).
  - 기간 조건은 "이슈 기간 내"로 표현.

### 4. 간결함·짧게

- 전체 공지는 짧게. 각 항목은 1~2문장이면 충분하다.
- 불필요한 수식어("매우", "상당히", "심각한", "very", "significantly", "serious") 제거.
- 한국어: "~하게 되었습니다"보다 "~되었습니다" / "~했습니다"로 줄인다.
- 영어: 수동태 남용하지 않기. 짧고 단정적인 문장 선호.
- 같은 뜻의 반복 제거.

## 번역 시 유의사항 (영어 버전 작성 시)

- **한국어 초안을 먼저 완성한 뒤 영어로 번역하는 방식을 권장**한다. 영어로 먼저 쓰고 한국어로 옮기면 한국어가 어색해진다.
- **번역 누락 절대 금지**: 모든 불릿, 모든 문장을 빠짐없이 번역한다. 특히 마지막 항목(조치 내용/Resolution)과 마무리 문구가 누락되지 않도록 최종 점검한다.
- 고정 번역(위 라벨 대응표)은 그대로 따른다. 임의 변경 금지.
- 영어로 옮길 때 자연스러움을 위해 약간 의역은 허용되나, **정보 누락/추가는 금지**한다.
- Affected Scope이 구체적일 때: "이슈 기간 내 새롭게 생성된 유저챗" → "user chats newly created during the issue period"처럼 직역에 가깝게.
- 한국어에만 있는 문화적 뉘앙스(예: "~드립니다" 같은 공손체)는 영어의 정중한 비즈니스 톤으로 자연스럽게 옮긴다.

## 작성 절차

1. **용어 확인**: 위 채널톡 공식 용어집을 참고해 이번 장애가 어떤 기능에 해당하는지, 정식 표기는 무엇인지 확인한다.

2. **입력 읽기**: 텍스트든 이미지든, 다음 항목을 먼저 뽑아낸다.
   - 영향받은 기능/제품 (사용자 관점 이름으로)
   - 발생 시간 (시작/종료/지속 시간)
   - 영향 범위 (누가, 어떤 조건에서 겪었는지)
   - 원인 (내부 기술 용어 포함된 상태로 일단 추출)
   - 조치 내용
   - 재발 방지 대책

3. **변환 (한국어 먼저)**: 각 항목을 위 4가지 원칙에 따라 변환한다. 특히 **원인**과 **영향 범위**에서 기술 용어가 남아있지 않은지, 채널톡 공식 용어가 정식 표기(예: "알프" → "ALF")로 바뀌어 있는지 점검한다.

4. **증상 재구성**: 원문이 "원인 중심"으로 쓰여 있다면, 사용자 관점 증상을 유추해서 "발생 현상" 블록 또는 한국어 첫 문단에 넣는다. 원문에 증상이 명시되어 있으면 그걸 1순위로 쓴다.

5. **한국어 조립**: 한국어 템플릿에 채워 넣는다.

6. **영어 번역**: 완성된 한국어 버전을 영어로 옮긴다. 라벨 대응표와 고정 표현을 따른다.

7. **최종 점검** (양쪽 버전 모두):
   - 내부 전용 용어/모델명/컴포넌트명이 남아있지 않은가?
   - 채널톡 공식 용어가 정식 표기로 쓰였는가? ("알프" → **ALF**, 유저챗/태스크 등은 그대로 유지)
   - 사과 문구가 있는가?
   - 한국어와 영어 버전의 항목 수가 일치하는가? (번역 누락 체크)
   - 한국어 문장이 영어 버전에 실수로 섞여 있지 않은가?
   - 사용자가 "내가 뭘 겪었던 건지"를 이 글만 보고 이해할 수 있는가?
   - 길이가 과하지 않은가? (각 버전 대략 5~10줄 이내)

## 출력 시 유의사항

- 한국어 버전과 영어 버전을 **각각 코드 블록(```)으로 감싸** 보여준다. (복사 붙여넣기 편의)
- 한국어 버전을 먼저, 그 다음 영어 버전을 보여준다. (제품팀이 한국어를 먼저 검토하는 흐름)
- **서두/말미 메타 멘트를 절대 넣지 않는다.** 아래 멘트들은 모두 출력 금지:
  - ❌ "안녕! 스킬을 먼저 확인해볼게."
  - ❌ "용어집도 확인할게."
  - ❌ "Incident-status-writer 스킬 읽기" 같은 진행 상황 안내
  - ❌ "조정이 필요한 부분이 있으면 말씀해주세요" 같은 마무리 멘트
  - ❌ "아래는 공지 내용입니다:" 같은 도입부
  - ❌ 원문을 요약하거나 변환 근거를 설명하는 말
- **첫 메시지부터 바로 한국어 코드블록**으로 시작한다. 코드블록 두 개 외에는 아무것도 출력하지 않는 것이 기본이다.
- 스킬 파일/용어집 확인은 내부적으로 수행하되, 사용자에게는 그 과정을 말하지 않는다.
- 원본에 재발 방지 대책이 빠져있거나 불명확하면, 임의로 지어내지 말고 해당 위치에 `[재발 방지 대책이 원문에 없어 비워뒀습니다 / Preventive measures not specified in the original report]` 식으로 표시해서 사용자가 채울 수 있게 한다.

### 올바른 출력 예시 (이 구조 그대로)

사용자가 장애 내용을 주면, **즉시 이렇게만** 출력한다:

````
```
- 발생 시간: ...
- 영향 범위: ...
- 원인: ...
- 조치 내용: ...
```

```
- Incident Time: ...
- Affected Scope: ...
- Cause: ...
- Resolution: ...
```
````

위아래 아무 말도 붙이지 않는다. 사용자가 추가 질문이나 수정 요청을 하면 그때 대응한다.

## 예시

### 입력 (제품팀 내부 공유)
```
@캐롤 공유드립니다!
- 발생 시간: 2026. 04. 22 21:30 ~ 2026. 04. 23 12:00 (KST)
- 영향 범위: Claude sonnet 4.6 모델 사용하는 채널, 이슈 기간 내에 새롭게 생성된 유저챗 또는 알프와 대화한 유저챗
- 원인: Claude sonnet 4.6 모델에서 tool call 관련 히스토리 처리 로직 변경 후 발생
- 조치 내용: 해당 로직 롤백
- 재발 방지 대책: 내부 회귀 테스트 강화
```

### 출력 (한국어)
```
- 발생 시간: 2026-04-22 21:30 ~ 2026-04-23 12:00 KST (약 14시간 30분간)
- 영향 범위: 이슈 기간 내 새롭게 생성되어 태스크가 동작한 유저챗
- 원인: 시스템의 특정 처리 로직 변경 후 일시적으로 응답이 원활하지 않은 현상 발생
- 조치 내용: 해당 로직을 이전 상태로 복구하였으며, 동일한 문제가 발생하지 않도록 내부 점검 절차를 강화하였습니다.

동일한 불편을 겪으시지 않도록 모니터링을 강화하겠습니다.
```
